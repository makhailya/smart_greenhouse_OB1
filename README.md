# 🌱 Умная теплица (Smart Greenhouse)

Дипломный проект **OB1 — Проект самообучения**

Программная симуляция автоматизированной теплицы: система читает показания датчиков, принимает решения и управляет исполнительными устройствами в реальном времени через графический интерфейс.

---

## 📋 Описание

Приложение моделирует работу умной теплицы без физического оборудования. Три виртуальных датчика поставляют случайные, но реалистичные показания, на основе которых контроллер автоматически включает или выключает три устройства:

- 💧 **Полив** — включается, если влажность почвы ниже порога, а также по расписанию (08:00, 12:00, 18:00)
- 🌬️ **Проветривание** — окно открывается, если температура воздуха превышает порог
- 💡 **Освещение** — фитолампа включается, если освещённости недостаточно

Все события и показания датчиков сохраняются в базу данных SQLite для последующего анализа.

## ⚙️ Конфигурация

Пороговые значения автоматизации задаются в `config.json`:

```json
{
  "thresholds": {
    "humidity_min": 40,
    "temperature_max": 30,
    "light_min": 300
  },
  "schedule": {
    "watering_times": ["08:00", "12:00", "18:00"]
  }
}
```

Значения можно менять без изменения кода программы.

## 🚀 Установка и запуск

### Требования
- Python 3.11+
- [Poetry](https://python-poetry.org/)

### Установка зависимостей

```bash
git clone <ссылка-на-репозиторий>
cd smart_greenhouse
poetry install
```

### Запуск приложения

```bash
poetry run python main.py
```

После запуска откроется окно с панелями датчиков, статусами устройств и журналом событий. Интерфейс обновляется автоматически каждые 3 секунды.

## 🖥️ Возможности интерфейса

- Просмотр текущих показаний датчиков в реальном времени с цветовой индикацией (зелёный — норма, красный — отклонение от порога)
- Просмотр статуса каждого устройства (включено/выключено)
- Ручное управление устройствами через кнопки
- Журнал последних 20 событий с указанием времени и причины срабатывания

---

## 🧱 Архитектурные решения

- **Наследование** — датчики (`Sensor`) и устройства (`Actuator`) построены через базовый класс с переопределением поведения в дочерних классах, что упрощает добавление новых типов датчиков или устройств
- **Разделение ответственности** — каждый модуль решает одну задачу: `sensors.py` ничего не знает о базе данных, `gui.py` ничего не знает о пороговых значениях
- **Хранение конфигурации отдельно от кода** — пороги вынесены в `config.json`
